Dynamic Programming

DP is just memoized search: a recursion that remembers its answers so it never solves the same subproblem twice. Master one worked example — the number triangle — and the four-part recipe behind every DP falls out of it.

The big picture

What dynamic programming really is

Two sentences carry the whole topic:

Essence

DP = memoized search. Take a divide-and-conquer recursion and cache each subproblem's answer in a table.

Why it's fast

It eliminates repeated computation. The same subproblem is solved once, not exponentially many times.

“Why is DP fast? Because I avoid repeated computation — I never compute the same state many times. That is the whole reason.” — the instructor [33:42]

This is also the hardest week. As the instructor likes to contrast: BFS you can learn in a day; DP you might study for months and still not finish. The good news is that almost every interview DP reduces to a small recipe you can drill.

One problem, four solutions

The number triangle

Given a triangle of numbers, walk from the top to the bottom (each step goes to the cell directly below-left or below-right) and return the minimum path sum. We solve it four ways — the progression is the lesson.

① DFS — traverse (enumerate every path)

Carry the running sum down each path; record the best at the bottom.

Python · O(2ⁿ) 
def minimum_total(triangle):
    n = len(triangle)
    best = [float("inf")]

    def traverse(x, y, path_sum):
        if x == n:                       # walked past the bottom
            best[0] = min(best[0], path_sum)
            return
        traverse(x + 1, y,     path_sum + triangle[x][y])
        traverse(x + 1, y + 1, path_sum + triangle[x][y])

    traverse(0, 0, 0)
    return best[0]

Every cell branches into two, so the path count doubles each row → O(2ⁿ).

② DFS — divide & conquer (return the answer)

Instead of carrying state down, ask each cell: “what is the best sum from me to the bottom?”

Python · O(2ⁿ) 
def divide_conquer(x, y):                # best sum from (x,y) to the bottom
    if x == n:
        return 0
    left  = divide_conquer(x + 1, y)
    right = divide_conquer(x + 1, y + 1)
    return min(left, right) + triangle[x][y]
# answer = divide_conquer(0, 0)

Cleaner, but still O(2ⁿ) — it recomputes shared cells like (2,1) over and over.

③ Divide & conquer + memoization — the leap

One cache turns the exponential recursion into a polynomial one. This is dynamic programming, written as top-down search.

Python · O(n²) 
def minimum_total(triangle):
    n = len(triangle)
    memo = {}

    def dfs(x, y):
        if x == n:
            return 0
        if (x, y) in memo:                 # already solved → reuse
            return memo[(x, y)]
        memo[(x, y)] = min(dfs(x + 1, y),
                            dfs(x + 1, y + 1)) + triangle[x][y]
        return memo[(x, y)]

    return dfs(0, 0)

There are only ~n²/2 distinct (x, y) states and each is computed once → O(n²).

Why a binary tree needs no memoization but a triangle does The instructor's sharpest point [34:01]: a binary-tree recursion never benefits from a cache because the left and right subtrees never reconverge — you can't reach the same node two ways. In the number triangle you can (go right-then-left vs left-then-right both land on (2,1)), so the same subproblem repeats. Overlapping subproblems = the precondition for DP. No overlap, no speedup.

④ Bottom-up multi-loop DP

The same recurrence, filled iteratively from the bottom row up — no recursion at all.

Python · O(n²) 
def minimum_total(triangle):
    n = len(triangle)
    # f[y] = best sum from the current row's cell y down to the bottom
    f = triangle[n - 1][:]                  # initialize with the bottom row
    for x in range(n - 2, -1, -1):       # work upward
        for y in range(x + 1):
            f[y] = min(f[y], f[y + 1]) + triangle[x][y]
    return f[0]
Top-down memoized search vs bottom-up loops
Bottom-up loopsTop-down memoized search
ProsConventional; most interviewers expect it; allows space optimization (rolling array).Mechanical to derive straight from a search; sometimes skips unreachable states, saving time.
ConsRequires thinking out the fill order.Recursion (stack depth / overhead).
Bottom-up and top-down have no big advantage over each other — one thinks forward, the other backward. (The course standardizes on the top-down direction for teaching.)
Pattern recognition

When to use DP — and when not to

You usually can't be told “this is a DP problem” — you infer it from the question's shape.

Likely DP if it asks for one of…

  1. a maximum or minimum value,
  2. whether something is feasible (yes/no), or
  3. the number of ways (count of plans).

Very unlikely DP if…

  1. it wants all concrete plans, not a count (that's DFS — e.g. Palindrome Partitioning),
  2. the input is a set, not a sequence (e.g. Longest Consecutive Sequence), or
  3. brute force is already polynomial.
What DP is good at DP shines at collapsing exponential brute force (2ⁿ, n!) down to polynomial (n², n³). It is not the tool for shaving n³ down to n² — don't reach for it there.
The recipe

The four elements of DP

Every DP is four decisions, and they map one-to-one onto the three elements of recursion:

DP elementWhat it isRecursion analogue
StateWhat a table entry f[…] means. Needs inspiration — it's the creative step.Definition (the params & return value)
Function (transition)How a bigger state is computed from smaller ones.The recursive decomposition
InitializationThe smallest/base states — the starting point. The recursion's exit / base case
AnswerThe largest state — what you finally return. (the top-level call's result)
“Once you've nailed the state — say f[i][j] = the best path from (i, j) to the bottom — everything after it is easy: the transition almost writes itself, because from (i, j) you can only step two ways.” — the instructor [57:05]

Finding the state is the hard, creative part; the other three usually follow mechanically once the state is right.

This week covers the two most common entry-level types: coordinate and sequence DP.

Type 1 of 2

Coordinate DP

The state is a position. f[x] or f[x][y] = the best/optimal value of getting from the start to coordinate x (or x, y). The transition looks at the one step before reaching that cell; initialization is the start; the answer is the end.

Minimum Path Sum

· LintCode

State f[i][j] = min cost from (0,0) to (i,j). Transition f[i][j] = min(f[i-1][j], f[i][j-1]) + grid[i][j].

Python · O(m·n) 
def min_path_sum(grid):
    m, n = len(grid), len(grid[0])
    f = [[0] * n for _ in range(m)]
    f[0][0] = grid[0][0]
    for i in range(1, m):                  # initialize column 0
        f[i][0] = f[i - 1][0] + grid[i][0]
    for j in range(1, n):                  # initialize row 0
        f[0][j] = f[0][j - 1] + grid[0][j]
    for i in range(1, m):
        for j in range(1, n):
            f[i][j] = min(f[i - 1][j], f[i][j - 1]) + grid[i][j]
    return f[m - 1][n - 1]
The Qi-Breaking Form — initializing a 2-D DP A reliable rule of thumb the instructor gives for any 2-D coordinate DP: when you initialize, initialize row 0 and column 0 first. Those are the cells with no “up” or “left” neighbor, so the general transition can't reach them — they must be seeded by hand.

Unique Paths

· LintCode

State f[i][j] = number of distinct paths to (i,j). Transition (look at the last step) f[i][j] = f[i-1][j] + f[i][j-1]. Row 0 and column 0 each have exactly one path → initialize to 1.

Python · O(m·n) 
def unique_paths(m, n):
    f = [[1] * n for _ in range(m)]   # row 0 & col 0 already 1
    for i in range(1, m):
        for j in range(1, n):
            f[i][j] = f[i - 1][j] + f[i][j - 1]
    return f[m - 1][n - 1]

Climbing Stairs

· LintCode

1-D coordinate DP. f[i] = number of ways to reach step i; you arrive from one or two steps below → f[i] = f[i-1] + f[i-2] (Fibonacci).

Python · O(n) 
def climb_stairs(n):
    if n <= 1:
        return 1
    f = [0] * (n + 1)
    f[0], f[1] = 1, 1
    for i in range(2, n + 1):
        f[i] = f[i - 1] + f[i - 2]
    return f[n]
Note on Jump Game I / II They look like coordinate DP, but the optimal solution is greedy — DP works only because the data ranges were lowered. Recognize when a cheaper technique dominates.
Type 2 of 2

Sequence (chaining) DP

A special case of coordinate DP where the “coordinate” is an index into a sequence and each state chains onto an earlier index that satisfies some condition.

Longest Increasing Subsequence (LIS)

· LintCode

The instructor's picture: think of the n numbers as n wooden stakes. Moving front-to-back and low-to-high, how many stakes can you step on? State f[i] = the longest increasing run that ends at stake i. Transition f[i] = max(f[j] + 1) over all j < i with nums[j] < nums[i].

Python · O(n²) 
def longest_increasing_subsequence(nums):
    if not nums:
        return 0
    n = len(nums)
    f = [1] * n                          # every stake is a run of length 1
    for i in range(n):
        for j in range(i):
            if nums[j] < nums[i]:           # can chain i onto j
                f[i] = max(f[i], f[j] + 1)
    return max(f)

The answer is max(f) — the longest run can end at any stake, not just the last.

DP records only ONE optimal value — not all optimal plans A DP table stores the best value for each state, so it can reconstruct one optimal solution, but it cannot enumerate all optimal solutions. The instant a problem asks for every optimal plan, you're back in DFS territory — this is the same boundary as the “all concrete plans ⇒ not DP” rule above.

More chaining practice: Perfect Squares, Largest Divisible Subset, Russian Doll Envelopes, Frog Jump.

Conclusion

The DP checklist

  1. DP = memoized search; it's fast because it kills repeated computation.
  2. Precondition: overlapping subproblems (no overlap ⇒ no benefit, e.g. a binary tree).
  3. Smells like DP when it asks for max/min, feasibility, or a count of ways.
  4. Not DP when it wants all concrete plans, the input is a set not a sequence, or brute force is already polynomial.
  5. Write any DP as four parts: State · Function · Initialization · Answer. State is the creative step.
  6. 2-D coordinate DP: initialize row 0 and column 0 (the Qi-Breaking Form).
  7. A DP table yields one optimal value, never all optimal plans.
