Depth First Search

When a problem asks you to produce every plan, it's almost always DFS — and almost every DFS problem is either a combination or a permutation. Learn the two backtracking templates and most search interviews collapse into them.

When to reach for DFS

The two-line rule

“When a problem asks you to find all the plans, it is DFS. And 90% of DFS problems are either a permutation or a combination.” — the Whip-Breaking Form (the instructor's rule of thumb)

Contrast with last week: BFS answers “how far / how many layers / shortest path”; DFS answers “enumerate every solution.” If the output is a list of concrete arrangements rather than a count or a distance, you're in DFS territory.

Combination

Elements are order-independent — {1,2} and {2,1} are the same. Running time scales with 2ⁿ. Canonical problem: Subsets.

Permutation

Elements are order-dependent — [1,2] and [2,1] differ. Running time scales with n!. Canonical problem: Permutations.

The template behind both

The three elements of recursion

Unless the interviewer forbids it, implement DFS with recursion. Every recursion is three decisions:

ElementQuestion it answers
DefinitionWhat parameters does the function take, what does it do, what does it return?
DecompositionHow do I shrink the parameters — make the problem smaller — and recurse?
ExitWhen can I return directly (the base case)?

The backtracking shape is always the same: choose → recurse → un-choose (append to the path, recurse, then pop). The path is the current partial plan; recording it at the right moment yields the answer set.

Family 1 of 2

Combination search

Subsets — the base template

· LintCode
Python 
def subsets(nums):
    nums.sort()
    results = []

    def dfs(start, path):
        results.append(list(path))        # every node is a valid subset
        for i in range(start, len(nums)):
            path.append(nums[i])
            dfs(i + 1, path)            # i+1 → each number used once, order fixed
            path.pop()                     # backtrack (un-choose)

    dfs(0, [])
    return results

Combination Sum

· LintCode

Three differences from Subsets — and each maps to one small code change:

Difference from SubsetsCode change
The combination's sum is capped at a targetCarry a remain parameter; record when it hits 0, prune when it goes negative
The input may contain duplicatesDedup the input first
A number may be reused many timesRecurse from i, not i+1
Python 
def combination_sum(candidates, target):
    candidates = sorted(set(candidates))    # dedup the input
    results = []

    def dfs(start, remain, path):
        if remain == 0:
            results.append(list(path))
            return
        for i in range(start, len(candidates)):
            if candidates[i] > remain:          # sorted → all later are bigger too
                break
            path.append(candidates[i])
            dfs(i, remain - candidates[i], path)  # i → reuse the same number
            path.pop()

    dfs(0, target, [])
    return results

Combination Sum II & Palindrome Partitioning

· dedup & pruning variants

Combination Sum II — each number used at most once, but the input has duplicates and duplicate combinations must be avoided. Sort, recurse from i+1, and skip a duplicate at the same depth:

Python · the dedup line 
        for i in range(start, len(candidates)):
            if i > start and candidates[i] == candidates[i - 1]:
                continue                       # skip duplicate choice at this level
            ...
            dfs(i + 1, remain - candidates[i], path)  # i+1 → use once

Palindrome Partitioning — partition a string so every piece is a palindrome. Same combination skeleton; at each step try every prefix that is a palindrome, then recurse on the rest. (And note: because it returns all concrete partitions, it's DFS, not DP.)

The dedup mantra Two distinct dedup needs: (1) duplicate numbers in the input → sort + “skip if same as previous at this depth”; (2) reusing vs not reusing a number → recurse from i vs i+1. Keep them straight and every combination variant is the same template.
Family 2 of 2

Permutation search

Order matters, so there's no start index — instead a visited array marks which elements are already in the current path.

Permutations

· LintCode
Python 
def permute(nums):
    results = []
    visited = [False] * len(nums)

    def dfs(path):
        if len(path) == len(nums):       # a full arrangement
            results.append(list(path))
            return
        for i in range(len(nums)):
            if visited[i]:
                continue
            visited[i] = True
            path.append(nums[i])
            dfs(path)
            path.pop()
            visited[i] = False

    dfs([])
    return results

Permutations II — dedup

· LintCode

Sort, then skip a duplicate value whose identical predecessor hasn't been used at this level — this fixes a canonical order among equal elements so each multiset permutation appears once.

Python · the dedup guard 
            if visited[i]:
                continue
            if i > 0 and nums[i] == nums[i - 1] and not visited[i - 1]:
                continue                       # equal value, predecessor unused → skip

Analysis

The universal DFS complexity formula

“The general time complexity of a depth-first search is O(number of answers × time to construct each answer).” — the instructor [43:40]
Problem# answersPer-answerTotal
Subsets2ⁿO(n)O(n · 2ⁿ)
Permutationsn!O(n)O(n · n!)
General searchS (can't be closed-form)O(n)O(S · n)
Interview move When asked the complexity of a search problem, first figure out how many answers there are. If you can't express it as a clean 2ⁿ or n!, just call the count S and answer O(S · n). That's an accepted, honest answer.
Two more arenas

Search in a graph · non-recursive DFS

Word Ladder & Word Ladder II

Word Ladder asks for the shortest transformation length — so the shortest-path part is actually BFS, not DFS. Word Ladder II wants all shortest transformation sequences, the classic hybrid: BFS to compute distances from the end, then DFS to reconstruct every shortest path (only stepping to neighbors with strictly smaller distance). A great reminder that “find all plans” = DFS, but “shortest” = BFS.

Non-recursion: DFS with an explicit stack

If recursion is disallowed, simulate it with a stack. The course flags the iterative tree traversals as must-memorize programs.

Python · iterative preorder 
def preorder(root):
    if not root:
        return []
    stack, result = [root], []
    while stack:
        node = stack.pop()
        result.append(node.val)
        if node.right:                  # push right first so left is popped first
            stack.append(node.right)
        if node.left:
            stack.append(node.left)
    return result

Expression Expand

Decode strings like 3[a2[c]] with a stack — push contexts on [, pop and repeat on ].

Flatten Nested List Iterator

Design question: should the real work live in has_next or next? Put it in has_next (lazily advance the stack to the next integer) so next is trivial and idempotent.

Must-practice stack problems: Implement Queue by Two Stacks, Largest Rectangle in Histogram, Min Stack.

Conclusion

The DFS checklist

  1. “Find all plans” ⇒ DFS. (The Whip-Breaking Form.)
  2. Almost every DFS is a combination (order-independent, ~2ⁿ) or a permutation (order-dependent, ~n!).
  3. Implement with the three elements of recursion; the loop body is always choose → recurse → un-choose.
  4. Combinations use a start index; permutations use a visited array.
  5. Reuse a number ⇒ recurse from i; use once ⇒ from i+1. Dedup inputs ⇒ sort + skip equal-at-same-depth.
  6. Complexity = O(#answers × per-answer time); when unsure, O(S · n).
  7. No recursion allowed ⇒ explicit stack; memorize the iterative tree traversals.
