Binary Search

One bulletproof template, three levels of mastery. Halve the search space each step for O(log n) — but the real skill is recognizing the problems that don't look like search at all and binary-searching the answer.

Why O(log n)

Complexity is a hint

Binary search is T(n) = T(n/2) + O(1) = O(log n): O(1) work shrinks the problem by half. In interviews, the target complexity often tells you the algorithm:

ComplexityUsually means…
O(log n)almost always binary search
O(√n)factoring / primes
O(n)common — a single scan
O(n log n)probably involves sorting
O(n²), O(n³)arrays, enumeration, DP
O(2ⁿ) / O(n!)combination / permutation search
“Anything better than O(n) is almost only ever O(log n) binary search. Reverse-engineering the algorithm from the required complexity is a standard interview move.” — the Sword-Breaking Form (the instructor's rule of thumb)
The one template

A binary search that never loops forever

The three classic pain points: infinite loops, which end condition? (start <= end / start < end / start + 1 < end), and which pointer move? (start = mid / mid + 1 / mid - 1). The course's template removes all three by fixing four things:

1 · Loop while start + 1 < end

Stop when the two pointers are adjacent — so they never collide and spin.

2 · mid = start + (end − start) // 2

Overflow-safe midpoint (matters in Java's fixed-width ints).

3 · Compare nums[mid] to target

Move start or end to mid — never mid±1.

4 · Check nums[start] & nums[end]

After the loop exactly two candidates remain — test both.

Python · universal template (first position of target) 
def binary_search(nums, target):
    if not nums:
        return -1
    start, end = 0, len(nums) - 1
    while start + 1 < end:                  # leave two candidates → no infinite loop
        mid = start + (end - start) // 2     # overflow-safe
        if nums[mid] == target:
            end = mid                          # first position → keep the left half
        elif nums[mid] < target:
            start = mid                        # never mid+1 — the template forbids it
        else:
            end = mid
    if nums[start] == target:                  # check the two survivors, left first
        return start
    if nums[end] == target:
        return end
    return -1
Last position? For the last position, change only the equality branch to start = mid, and check nums[end] before nums[start]. Same skeleton, mirror the tie-breaking.
Level 1 of 3

Binary search on index — OOXX

The array splits into a run of “O” (condition false) then a run of “X” (condition true). You hunt the boundary — the first X (or last O).

Python · First Bad Version 
def find_first_bad_version(n):
    start, end = 1, n
    while start + 1 < end:
        mid = start + (end - start) // 2
        if is_bad_version(mid):
            end = mid                  # bad → answer is mid or earlier (an X)
        else:
            start = mid                # good → answer is later (an O)
    return start if is_bad_version(start) else end

Same shape: Search in a Big Sorted Array (exponentially probe to find a bound, then binary search) and Find Minimum in Rotated Sorted Array (OOXX against the last element — note: compare to the last number, not the first).

Level 2 of 3

Binary search on index — half-half

No clean OOXX boundary exists, but at each mid you can still decide which half must contain a solution and discard the other.

Python · Find Peak Element 
def find_peak(nums):
    start, end = 0, len(nums) - 1
    while start + 1 < end:
        mid = start + (end - start) // 2
        if nums[mid] < nums[mid + 1]:
            start = mid                # still ascending → a peak lies to the right
        else:
            end = mid                  # descending → a peak is here or to the left
    return start if nums[start] > nums[end] else end

Search in Rotated Sorted Array

· “master this and you can say you know binary search”

At each step one half is sorted — figure out which, then test whether the target lies inside that sorted half.

Python 
def search_rotated(nums, target):
    if not nums:
        return -1
    start, end = 0, len(nums) - 1
    while start + 1 < end:
        mid = start + (end - start) // 2
        if nums[mid] == target:
            return mid
        if nums[start] < nums[mid]:               # left half is sorted
            if nums[start] <= target <= nums[mid]:
                end = mid
            else:
                start = mid
        else:                                    # right half is sorted
            if nums[mid] <= target <= nums[end]:
                start = mid
            else:
                end = mid
    if nums[start] == target:
        return start
    if nums[end] == target:
        return end
    return -1
Level 3 of 3 · the real test

Binary search on the answer

No array is handed to you, and the problem doesn't look like search at all. The tell: it asks for the maximum or minimum value satisfying some condition, and that condition is monotonic (if value v works, everything below/above it works too). Binary-search the answer space and test feasibility at each guess.

Sqrt(x)

· last n with n² ≤ x
Python 
def sqrt(x):
    if x < 2:
        return x
    start, end = 1, x
    while start + 1 < end:
        mid = start + (end - start) // 2
        if mid * mid <= x:
            start = mid                # feasible → try a larger root
        else:
            end = mid
    return end if end * end <= x else start

Wood Cut

· biggest length giving ≥ k pieces

Binary-search the cut length L; longer L → fewer pieces (monotonic), so search for the largest L that still yields ≥ k pieces.

Python 
def wood_cut(woods, k):
    if not woods:
        return 0
    start, end = 1, max(woods)
    while start + 1 < end:
        mid = start + (end - start) // 2
        if sum(w // mid for w in woods) >= k:   # enough pieces at length mid
            start = mid                            # try longer
        else:
            end = mid                              # too long → shorten
    return end if sum(w // end for w in woods) >= k \
               else (start if sum(w // start for w in woods) >= k else 0)

Copy Books is the same idea: “split the array into k subarrays minimizing the largest subarray sum” — binary-search that largest sum.

Interview judgment

Recursion or while-loop?

Binary search works either way. Whether to use recursion in an interview comes down to four questions — and the meta-rule: don't decide silently, discuss it with the interviewer.

  1. Did the interviewer forbid recursion? (If unsure, ask.)
  2. Would avoiding recursion make the code much more complex?
  3. Could the recursion depth get very deep (stack-overflow risk)?
  4. Is the point of the question recursion-vs-iteration, or just whether you can recurse?
Bonus: Rotate Array — the three-step reversal To rotate, reverse in three moves: reverse the first part, reverse the second part, reverse the whole. [4,5,1,2,3] → [5,4,1,2,3] → [5,4,3,2,1] → [1,2,3,4,5]. Powers Recover Rotated Sorted Array and Rotate String.
Conclusion

The Binary Search checklist

  1. Better than O(n) ⇒ think O(log n) binary search (the Sword-Breaking Form).
  2. The four-point template: start + 1 < end, mid = start + (end−start)//2, compare nums[mid], then check both survivors. Move pointers to mid, never mid±1.
  3. Realm 1 (OOXX): find the first/last position meeting a condition.
  4. Realm 2 (half-half): discard the half with no solution (Find Peak, Rotated Array).
  5. Realm 3 (on the answer): doesn't look like search — binary-search the value space when feasibility is monotonic (Sqrt, Wood Cut, Copy Books).
  6. Recursion vs loop: judge by the four questions, and discuss with the interviewer.
