Breadth First Search

One queue, one template, three arenas — binary trees, graphs, and matrices. BFS is the workhorse for graph traversal and shortest path in a simple graph, and the single most reliable pattern to reach for in interviews.

The big picture

When should you use BFS?

Almost every BFS interview problem is one of just two shapes. If you can classify the problem into one of these, the template writes itself.

① Graph traversal

Visit every reachable node, layer by layer.

Level order Connected component Topological sort

② Shortest path

Fewest steps from A to B — only in a simple graph.

Every edge length = 1 Undirected / unweighted
“Pretty much every company has at least one BFS question, many have more than one — so its priority is high. The good news: it's not hard. Willing to study, you can learn it in a single day. Compare that to dynamic programming — you could study three months, a year, and still not finish.” — the instructor's opening pitch [00:43–01:43]
Precondition for shortest path BFS gives the shortest path only in a simple graph: every edge has length 1 and there are no weights/directions that change cost. The moment edges have different weights, you need Dijkstra / SPFA / DP instead — not plain BFS.
“If a problem can be solved with BFS, do not solve it with DFS.” — the Spear-Breaking Form (the instructor's rule of thumb)

Why? BFS is iterative, so it can't blow the recursion stack on deep/large graphs, the layer structure gives shortest paths "for free," and the template is short and hard to get wrong. Reach for DFS only when the interviewer explicitly asks, or when the problem is fundamentally about enumerating all paths/permutations (that's next week).

The concrete reason from the lecture — stack overflow The instructor's argument [114:19–114:46]: if you DFS a big grid, the recursion depth can be as large as the whole board (imagine a snake-like region of all-1 cells, depth ≈ rows×cols). That deep recursion can overflow the call stack — a real crash, not just a style nit. BFS, being iterative on an explicit queue, never has this problem. “So whenever a problem can be solved with BFS, use BFS — that's best.”
The one template

BFS key points

1 · Queue is the core data structure

BFS uses a queue (FIFO). Thinking question from the deck: could you use a stack instead? You can still visit every node, but a stack is LIFO, so you lose the layer-by-layer order — meaning you lose shortest-path and level-order guarantees. A stack turns BFS into something DFS-shaped. So: queue, not stack.

2 · Do you need layers?

A layered BFS has one extra loop compared to an unlayered one. The trick is to snapshot the queue size at the start of each level:

from collections import deque

queue = deque([root])
while queue:
    size = len(queue)               # ← snapshot THIS level's count
    for _ in range(size):           # process exactly one layer
        node = queue.popleft()
        # ... record node, then enqueue its children/neighbors
    level += 1                      # one full layer done
Classic bug — the layer-size gotcha You must fix the layer count before draining it. range(len(queue)) reads the length once, so it's safe. The bug appears if you instead iterate the queue while mutating it (e.g. for node in queue: while append()-ing children), or re-read len(queue) inside the loop — it keeps growing as you enqueue, and you no longer cut a clean layer. Always snapshot size = len(queue) up front.
(In the original Java this is the famous int size = queue.size(); vs i < queue.size() trap.)

3 · Graphs need a visited set; trees don't

A tree has no cycles, so every node is reached exactly once. A general graph has cycles — the same node can be pushed into the queue more than once — so you need a set (or dict) to mark visited nodes. Mark visited at enqueue time, not dequeue time, or a node can be queued multiple times before it's first processed.

Mental model from the lecture — "a full BFS is three nested loops" The instructor's framing [120:33–121:13]: a complete layered graph BFS has exactly three loops, and you drop loops as the problem simplifies:
  1. Loop 1 — while queue: (queue not empty).
  2. Loop 2 — the layer: for _ in range(len(queue)). Drop this if you don't need levels.
  3. Loop 3 — expand neighbors: for nb in node.neighbors. On a binary tree you skip it and just touch node.left / node.right directly.
“Once you know what each of the three loops does, the rest is basically the same code every time.”
Arena 1 of 3

BFS in a binary tree

The friendliest arena: trees have no cycles, so no visited set needed. This is the purest expression of the template.

Binary Tree Level Order Traversal

· LintCode Level order

Return node values grouped by depth: [[A],[B,C],[D,E,F]]. This is the canonical layered BFS.

Python 
from collections import deque

def level_order(root):
    results = []
    if not root:
        return results

    queue = deque([root])
    while queue:
        size = len(queue)               # snapshot the layer
        level = []
        for _ in range(size):
            node = queue.popleft()
            level.append(node.val)
            if node.left:
                queue.append(node.left)
            if node.right:
                queue.append(node.right)
        results.append(level)
    return results
Three related problems (same template, tiny tweaks)
  • Level Order II — return layers bottom-up: build normally, then results.reverse() (or results[::-1]).
  • Zigzag Level Order — flip direction each layer: keep a boolean and insert into the level list from the front on odd layers.
  • Convert Binary Tree to Linked Lists by Depth — same loop, but stitch each layer into a linked list instead of a list.

Binary Tree Serialization

· LintCode Microsoft · Yahoo
“The first time I interviewed, I didn't know what serialization was either — I was interviewing at Google, got this question, and argued with the interviewer for ages, totally lost. A lot of schools never teach it, yet in industry it's genuinely very useful.” — the instructor [32:23–33:07]

Serialization = turning structured in-memory data into a string; deserialization is the reverse. The contract: a serializer must support a matching deserializer — if you can turn the object into a string but can't turn it back, it's just a fancy print and serves no purpose [33:34].

Serializeobject → string
Deserializestring → object

Why serialize at all?

  1. Persistence — RAM is volatile; to survive a power-off, structured data must be written to disk as bytes and rebuilt later.
  2. Network transport — two machines can't read each other's memory, so data is flattened to a string/byte stream, sent, and parsed back on the other side.

Common real-world formats: XMLJSON Thrift (Facebook)ProtoBuf (Google). Two design axes pull against each other:

  • Compression ratio — smaller is cheaper to store/transmit (Thrift, ProtoBuf optimize for this).
  • Readability — a human can read the data directly (JSON, LintCode's I/O format).
How LintCode serializes a tree Any scheme is valid as long as it round-trips. LintCode uses BFS order (level by level, with # for null children) — the nice side effect is that you can read the string and hand-draw the whole tree. e.g. {1,2,3,#,#,4,5}.
Arena 2 of 3

BFS in a graph

What changes vs a tree? A graph can contain cycles. A cycle means the same node can be reached along multiple paths and thus pushed into the queue repeatedly. The fix is one line of discipline: a set / dict of visited nodes.

Cycles in a graph — the cycle rule Without a visited set, a cycle makes BFS loop forever (or at least re-process nodes). Mark a node visited the moment you enqueue it, so it can never enter the queue twice.

How do you represent a graph?

Usually an adjacency list: dict[node, list[node]] (e.g. collections.defaultdict(list)) or, for integer-labelled nodes, a list[list[int]]. Each node maps to its neighbors. (An adjacency matrix [[False]*n for _ in range(n)] is the alternative — simpler but O(N²) space.)

Clone Graph

· LintCode Facebook Connected component

Deep-copy an undirected graph given one node. The clean BFS solution is three explicit passes — separating "discover the structure" from "rebuild it" is what keeps the code bug-free.

The instructor’s “one point reaches the whole area” picture “Connected component is: I give you one contact, and you have to find everyone in the whole school.” [05:22] Starting from one node, BFS fans out across every edge until the entire reachable component is discovered — that's step ① below.
Python 
from collections import deque

def clone_graph(node):
    if not node:
        return None

    # ① BFS: collect every reachable node
    nodes = get_nodes(node)

    # ② copy nodes (old → new), neighbors still empty
    mapping = {n: UndirectedGraphNode(n.label) for n in nodes}

    # ③ copy edges using the mapping
    for n in nodes:
        for nb in n.neighbors:
            mapping[n].neighbors.append(mapping[nb])
    return mapping[node]


def get_nodes(node):
    queue = deque([node])
    visited = {node}                       # mark at enqueue!
    while queue:
        head = queue.popleft()
        for nb in head.neighbors:
            if nb not in visited:
                visited.add(nb)
                queue.append(nb)
    return list(visited)

Graph Valid Tree

A graph of n nodes is a tree iff it has exactly n−1 edges and is fully connected. Check edge count first (cheap), then BFS from node 0 and confirm you reached all n nodes.

Search Graph Nodes

Find the nearest node with value == target. Plain BFS, no layering needed — the first match popped is already nearest (BFS dequeues in non-decreasing distance order). Follow-up: to find all nearest matches, switch to a layered BFS and collect every match in the first layer that has one.

Must-master · Topological sort

Topological Sorting

“Almost every company has a topological-sort interview question.” — the instructor, on why this one is non-negotiable

Given a directed graph (typically a DAG of dependencies), output an ordering where every node comes before all nodes it points to. The BFS solution is Kahn's algorithm, driven by in-degree (number of incoming edges).

The instructor's intuition — "unlocking levels" Think of it as a video game / course prerequisites [91:33–91:58]: nodes with in-degree 0 are the courses with no prerequisites, so you can take them first. Taking a course "removes" it from the graph, which unlocks the courses that depended only on it — their in-degree drops to 0 and they go into the queue. Repeat until everything is unlocked.

The algorithm in three moves

  1. 1 Count the in-degree of every node.
  2. 2 Enqueue all nodes with in-degree 0 (no prerequisites).
  3. 3 BFS: pop a node into the result, then for each neighbor decrement its in-degree; when a neighbor hits 0, enqueue it.
Python 
from collections import deque

def top_sort(graph):
    order = []
    indeg = {}

    # ① count in-degrees
    for node in graph:
        for nb in node.neighbors:
            indeg[nb] = indeg.get(nb, 0) + 1

    # ② all in-degree-0 nodes start the queue
    queue = deque(node for node in graph if node not in indeg)

    # ③ BFS, peeling off in-degree as we go
    while queue:
        node = queue.popleft()
        order.append(node)
        for nb in node.neighbors:
            indeg[nb] -= 1
            if indeg[nb] == 0:
                queue.append(nb)
    return order
Can you do topo sort with DFS? — yes, but… DFS post-order + reverse also works. But per the Spear-Breaking Form, prefer BFS: it's iterative (no stack overflow on big graphs), and it naturally detects cycles — if the output contains fewer than N nodes, the graph had a cycle (some nodes never reached in-degree 0). That cycle check is exactly what "Course Schedule" asks for.

The topo-sort problem family

BFS in a matrix

Matrix vs graph — complexity

GraphMatrix
SizeN nodes, M edgesN rows × M cols
EdgesM up to O(N²)N·M nodes, ~N·M·2 edges (4 per cell, each shared by 2)
BFS timeO(M) → worst O(N²)O(N·M)

A matrix is just a graph in disguise: each cell is a node with edges to its 4 (or 8) neighbors. The win is that the "edges" are implicit, so BFS is a clean O(rows × cols).

Two lecture tips for matrix BFS ① The grid is your hash table [118:44–119:06]. You rarely need a separate visited structure — a cell is identified by its (x, y) coordinate, so a 2-D visited array (or marking the grid in place) is the hash set. No dict of node objects needed.
② Use R and C, not N and M [115:02–115:21]. In graph notation N,M means "N nodes, M edges"; in a matrix it's "N rows, M cols" with ~N·M·2 edges — a different meaning. Naming the matrix dims R × C avoids the mix-up.

The direction arrays — deltaX / deltaY

Instead of writing four near-identical blocks for up/down/left/right, iterate over delta arrays. This is the single most reused matrix-BFS idiom.

Python · 4 directions 
DX = [0, 1, 0, -1]   # right, down, left, up
DY = [1, 0, -1, 0]
# new cell = (x + DX[i], y + DY[i]) for i in range(4)
8 directions (deck question: write these out) 
DX = [-1, -1, -1, 0, 0, 1, 1, 1]
DY = [-1, 0, 1, -1, 1, -1, 0, 1]
Always pair deltas with an in-bound check 
def in_bound(x, y, grid):
    return 0 <= x < len(grid) and 0 <= y < len(grid[0])

Number of Islands

· LintCode Connected component

Count connected blobs of 1s (4-directionally). Scan every cell; when you hit an unvisited land cell, that's a new island — BFS-flood it to mark the whole island visited, then keep scanning.

Zombie in Matrix

· LintCode Level order

Zombies spread to adjacent humans each day; return the days until everyone is infected (or −1 if some human is walled off). This is multi-source layered BFS: seed the queue with all zombies at once, then each while turn = one day. The number of layers = the answer.

Multi-source BFS — the instructor's "bite" picture “Put all the zombies into the queue as the BFS starting points. Then do a layered traversal — each layer is every current zombie taking one bite in the four directions; the newly-bitten humans become zombies and go into the queue. Which layer you finish = which day it is.” [119:56–120:28] You never BFS from each source separately; the layered expansion computes every cell's minimum time simultaneously.

Knight Shortest Path

· LintCode Simple-graph shortest path

Fewest knight moves from source to target on a grid. Pure shortest-path BFS — just use the 8 knight-move deltas instead of 4 orthogonal ones:

DX = [1, 1, -1, -1, 2, 2, -2, -2]
DY = [2, -2, 2, -2, 1, -1, 1, -1]
Follow-up: speed up? → Bidirectional BFS The deck poses “speed up?” but the lecture leaves it as an open exercise — so here is the standard answer: run BFS from both the source and the target, alternating, and stop when the two frontiers meet. Intuition: instead of one ball of radius r, you grow two balls of radius r/2 — roughly 2·b^(r/2) work instead of b^r, an exponential cut on a wide branching factor like the knight's 8.

Build Post Office II

· LintCode Simple-graph shortest path

Place one post office on an empty cell to minimize the total walking distance from every house. The key insight is which direction to BFS from:

Method 1 — from each empty cell ✗

Enumerate every candidate office location, BFS to all houses, sum distances, keep the min. Wasteful: there are usually far more empty cells than houses, so you run many BFS over the whole grid.

Method 2 — from each house ✓

BFS from each house, accumulating each empty cell's distance to that house into a dist[][] grid. After all houses, the best empty cell is the one with the smallest accumulated sum. Fewer sources → faster.

“The faster method is reverse thinking: there are usually few houses (and even if there are many, they just act as obstacles). BFS from each house, add its distance to every empty cell onto that cell's running total, then sweep the grid for the smallest total. Don't BFS from the empty cells.” — the instructor [128:36–129:38]. Method 1 (from empty cells) is correct but TLEs; this is what passes.
The transferable trick When you must aggregate distances between two sets of points, BFS from the smaller set and accumulate onto the larger set. Same idea powers many "min total distance" grid problems.

Decision flow

  • Need layers / shortest #steps / "min days"? → layered BFS, size = len(queue).
  • Just need to visit all / count components? → unlayered BFS + visited set.
  • Directed dependencies / ordering / cycle detection? → topo sort (in-degree BFS).
  • Grid? → deltaX/deltaY + inBound; sources may be one or many.
  • Source & target both known, branching wide? → consider bidirectional BFS.
Conclusion

The seven things to remember

  1. If BFS can solve it, don't use DFS (unless the interviewer insists). — the Spear-Breaking Form.
  2. BFS has exactly two use cases: graph traversal (connected component + level order) and shortest path in a simple graph.
  3. To go layer-by-layer, snapshot size = len(queue) before the inner loop (for _ in range(size)) — never mutate the queue while iterating it.
  4. Trees need no visited set; graphs do (cycles), and you mark visited at enqueue time.
  5. Topological sort is mandatory — in-degree BFS; output < N ⇒ cycle.
  6. Matrix BFS = deltaX / deltaY + inBound; multi-source when many starts.
  7. Shortest path needs a simple graph (edges = 1). Speed up wide searches with bidirectional BFS.
“Don’t use DFS when BFS works. The two BFS use cases: graph traversal and shortest path in a simple graph. Need layers? size = len(queue). Topological sort is a must. Direction arrays: deltaX, deltaY, inBound.” — Slide 30 conclusion (translated)
