Heap & Stack (advanced)

Two structures pushed to their hard cases: the heap for “the extreme of a changing set” (boundary-inward flooding, streaming medians), and the monotonic stack for “the first bigger/smaller neighbor” problems.

Frame

What each does best

A heap answers “what's the current min/max of this set?” as elements come and go. A stack shines two ways here: augmented (carry extra info, like Min Stack) and monotonic (maintain sorted order to find nearest bigger/smaller elements). A recurring advanced idea: a sliding window = one add + one delete, so window problems reduce to a structure that supports both efficiently.

The signature heap problem

Trapping Rain Water I → II

I (1-D) needs no heap — two opposite pointers, each capped by the smaller side's running max:

Python · Trapping Rain Water I (two pointers) 
def trap(heights):
    if not heights:
        return 0
    left, right = 0, len(heights) - 1
    left_max, right_max = heights[left], heights[right]
    water = 0
    while left < right:
        if left_max < right_max:           # the lower side bounds the water
            left += 1
            left_max = max(left_max, heights[left])
            water += left_max - heights[left]
        else:
            right -= 1
            right_max = max(right_max, heights[right])
            water += right_max - heights[right]
    return water
II (2-D) — why a heap, and why boundary-inward In 2-D, water escapes through the lowest point on the current boundary. So push all border cells into a min-heap and always process the lowest. Each popped cell floods its unvisited neighbors up to its own height, then the neighbor joins the boundary at max(boundary height, its own height). Always expanding from the lowest wall is exactly what the heap gives you — O(mn·log(mn)).
Python · Trapping Rain Water II (min-heap, outside-in) 
import heapq

def trap_rain_water_ii(heights):
    if not heights or not heights[0]:
        return 0
    m, n = len(heights), len(heights[0])
    visited = [[False] * n for _ in range(m)]
    heap = []
    for i in range(m):                  # seed the heap with all border cells
        for j in range(n):
            if i in (0, m - 1) or j in (0, n - 1):
                heapq.heappush(heap, (heights[i][j], i, j))
                visited[i][j] = True
    water = 0
    dirs = [(0, 1), (0, -1), (1, 0), (-1, 0)]
    while heap:
        h, x, y = heapq.heappop(heap)      # lowest wall on the boundary
        for dx, dy in dirs:
            nx, ny = x + dx, y + dy
            if 0 <= nx < m and 0 <= ny < n and not visited[nx][ny]:
                visited[nx][ny] = True
                water += max(0, h - heights[nx][ny])
                heapq.heappush(heap, (max(h, heights[nx][ny]), nx, ny))
    return water
Two heaps + lazy deletion

Data Stream & Sliding Window Median

Data Stream Median keeps a max-heap (smaller half) and a min-heap (larger half) balanced so the median is a heap top — O(log n) per add. Sliding Window Median adds the window twist: every step is one add + one delete, and a binary heap can't delete an arbitrary element. Two fixes:

Lazy deletion

Mark removed values in a hash map; skip them when they surface at a heap top.

HashHeap

A heap plus a map from value → its index, so you can delete any element in O(log n). Stands in for a TreeSet.

Python · Data Stream Median (two heaps) 
import heapq

class MedianFinder:
    def __init__(self):
        self.lo, self.hi = [], []          # max-heap (neg), min-heap

    def add(self, num):
        heapq.heappush(self.lo, -num)
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def median(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2
The decomposition that unlocks window problems Any sliding-window problem = (a) add one element + (b) remove one element. Solve those two operations on the right structure (two heaps, monotonic deque, hash map) and the window falls out.
Stack, three ways

Stack: save state, reverse, go iterative — and monotonic

Three reusable stack ideas: temporarily hold valid info (Min Stack), reverse order (Implement Queue by Two Stacks), and turn recursion iterative (Expression Expand decodes 2[a2[b]] → abbabb with a stack of contexts).

The headline advanced technique is the monotonic stack: to find, for each element, the first bigger/smaller element to its left or right, keep a stack whose values stay sorted — pop until the order holds, and the element you pop against is your answer.

Python · Largest Rectangle in Histogram (monotonic increasing stack) 
def largest_rectangle_area(heights):
    stack = []                             # indices, increasing heights
    best = 0
    for i in range(len(heights) + 1):
        h = heights[i] if i < len(heights) else 0
        while stack and heights[stack[-1]] >= h:
            top = stack.pop()
            left = stack[-1] if stack else -1
            best = max(best, heights[top] * (i - left - 1))
        stack.append(i)
    return best

Same monotonic-stack engine: Maximal Rectangle (run histogram per row) and Max Tree (build a tree where each node's parent is the nearest larger value).

Conclusion

The Heap & Stack checklist

  1. Heap = the extreme of a changing set. Trapping Rain Water II floods boundary-inward from the lowest wall via a min-heap.
  2. Streaming median = two balanced heaps; window median adds delete → lazy deletion or a HashHeap.
  3. Every sliding window decomposes into add one + remove one.
  4. Monotonic stack finds the nearest bigger/smaller neighbor — Largest Rectangle, Maximal Rectangle, Max Tree, all O(n).
  5. Know the per-language names; in Python reach for heapq and sortedcontainers.
