Data Structure

A data structure is just a collection plus the operations it supports efficiently. Pick the structure by the operations you need. This week: Stack, Hash, and Heap — what each guarantees in O(1) or O(log n), and the signature problems behind each.

Structure 1 of 3

Stack

LIFO. Beyond simulating DFS, two stack ideas recur in interviews: the augmented stack (carry extra info alongside each element) and the monotonic stack (keep elements in sorted order).

Min Stack

· augmented stack

Support min() in O(1) by pushing the running minimum on a parallel stack.

Python 
class MinStack:
    def __init__(self):
        self.stack = []
        self.min_stack = []                # running minimum at each depth

    def push(self, x):
        self.stack.append(x)
        m = x if not self.min_stack else min(x, self.min_stack[-1])
        self.min_stack.append(m)

    def pop(self):
        self.min_stack.pop()
        return self.stack.pop()

    def min(self):
        return self.min_stack[-1]

Largest Rectangle in Histogram

· the monotonic stack

Keep a stack of increasing bar heights. When a shorter bar arrives, pop taller bars and settle their maximal rectangle — each bar is pushed and popped once → O(n).

Python 
def largest_rectangle_area(heights):
    stack = []                             # indices, heights strictly increasing
    best = 0
    for i in range(len(heights) + 1):
        h = heights[i] if i < len(heights) else 0   # sentinel flushes the stack
        while stack and heights[stack[-1]] >= h:
            top = stack.pop()
            left = stack[-1] if stack else -1
            width = i - left - 1
            best = max(best, heights[top] * width)
        stack.append(i)
    return best

The same monotonic-stack idea underlies Maximal Rectangle and Max Tree.

Other must-know stack problems Expression Expand (decode 3[a2[c]]), Implement Queue by Two Stacks and Implement Stack by Two Queues, and the iterator family — Flatten Nested List Iterator, BST Iterator, Zigzag Iterator, Flatten 2D Vector. Design tip for iterators: put the real work in has_next (advance to the next element lazily) so next stays trivial.
Structure 2 of 3

Hash

O(1) insert / find / delete. HashSet stores keys only; HashMap stores key→value; a HashTable is the same idea (historically thread-safe in Java). The magic is the hash function.

What a hash function must do Map any key to a fixed, pseudo-random integer in [0, capacity−1]. “Fixed” so the same key always lands in the same bucket; “pseudo-random” so keys spread out and collisions stay rare. Famous ones: MD5, SHA-1, SHA-2. For strings, a polynomial rolling hash with a “magic number” (often 31 or 33) works well: 
def hash_code(s, capacity):
    code = 0
    for ch in s:
        code = (code * 31 + ord(ch)) % capacity   # 31 = the "magic number"
    return code

Collisions are unavoidable, so a table needs a strategy:

Open hashing (chaining)

Each bucket holds a linked list of colliding entries.

Closed hashing (probing)

On collision, probe the next free slot in the array.

Rehashing — when the table gets crowded Once the load factor is high, find/insert degrades. Rehash: allocate a larger array (e.g. 2×) and re-insert every key with the new capacity. This is the Rehashing problem. Other hash applications: Subarray Sum (prefix-sum → index map), Anagrams, Longest Consecutive Sequence, Copy List with Random Pointer.
The classic hash design

LRU Cache

O(1) get and set with eviction of the least-recently-used key. The trick is a HashMap + doubly linked list: the map gives O(1) lookup; the list keeps usage order so you can move/evict in O(1). (Java's LinkedHashMap is exactly this.)

Python 
class Node:
    def __init__(self, key=0, val=0):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.map = {}                          # key -> Node
        self.head, self.tail = Node(), Node()    # sentinels
        self.head.next, self.tail.prev = self.tail, self.head

    def _remove(self, node):
        node.prev.next, node.next.prev = node.next, node.prev

    def _add(self, node):                      # insert just before tail (newest)
        node.prev, node.next = self.tail.prev, self.tail
        self.tail.prev.next = node
        self.tail.prev = node

    def get(self, key):
        if key not in self.map:
            return -1
        node = self.map[key]
        self._remove(node); self._add(node)    # mark most-recently-used
        return node.val

    def set(self, key, value):
        if key in self.map:
            self._remove(self.map[key])
        node = Node(key, value)
        self.map[key] = node
        self._add(node)
        if len(self.map) > self.cap:            # evict eldest at head
            lru = self.head.next
            self._remove(lru)
            del self.map[lru.key]
Structure 3 of 3

Heap

A complete binary tree giving O(log n) add/remove and O(1) peek at the min (min-heap) or max (max-heap). In practice you use a priority queue. Building a heap from n items — Heapify — is O(n).

When a heap is the answer Anything about “the current best/extreme on a stream”: Ugly Number II, Top K Largest Numbers, K Closest Points, Merge K Sorted Lists/Arrays, Kth Smallest in a Sorted Matrix.

Data Stream Median — two heaps

· the signature heap trick

Keep a max-heap for the smaller half and a min-heap for the larger half, balanced in size. The median is then a heap top (or the average of the two tops) in O(1).

Python 
import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []                       # max-heap (store negatives): smaller half
        self.hi = []                       # min-heap: larger half

    def add(self, num):
        heapq.heappush(self.lo, -num)
        heapq.heappush(self.hi, -heapq.heappop(self.lo))   # shuffle to keep order
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def median(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2
When a heap isn't enough

TreeMap (balanced BST)

A heap gives you the min or max cheaply, but it can't delete or update an arbitrary element efficiently. When you need the extreme and arbitrary modify/delete — all in O(log n) — reach for a TreeMap (a balanced binary search tree; ordered keys).

Conclusion

The Data Structure checklist

  1. Pick a structure by the operations it makes cheap — that's the whole game.
  2. BFS ⇒ Queue, DFS ⇒ Stack. Never swap them (the Arrow-Breaking Form).
  3. Stack tricks: augmented (Min Stack) and monotonic (Largest Rectangle — O(n)).
  4. Hash = O(1) via a function mapping any key to [0, capacity−1]; handle collisions (chaining/probing) and rehash when crowded.
  5. LRU Cache = HashMap + doubly linked list (O(1) get/set/evict).
  6. Heap = O(log n) add/remove, O(1) peek; two heaps track a streaming median.
  7. Need extreme plus arbitrary delete/update ⇒ TreeMap, not a heap.
